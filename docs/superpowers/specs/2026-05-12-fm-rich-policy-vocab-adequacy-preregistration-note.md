# Does a medium-fidelity codified mortgage policy still leave a vocabulary gap? — pre-registration (#11)

**Date:** 2026-05-12. **Status:** pre-registration — committed (and OTS-stamped) *before* any experimental code touches data, per the project's pre-reg discipline. **Substrate:** Fannie Mae Single-Family Loan Performance, **2018Q1** (already extracted on disk at `data/fanniemae/2018Q1.csv`, pipe-delimited, 113-col unified CRT layout). **Companion to:** the extension-admitted-band result (`2026-05-12-extension-admitted-band-result-note.md`) and the routable-population result (`2026-05-12-routable-population-test-result-note.md`), whose corrected finding this test extends — *and questions* — by swapping the **thin demo (3–4 named features)** for a **medium-fidelity codified policy (~12–14 named features)** on a different substrate.

---

## 0. Why this test, in one paragraph

On LendingClub 36-month, the thin-demo policy (`thin_demo_hmda.yaml`: names `{fico_range_low, dti, annual_inc, emp_length}`) constructs policy-admissible refinement bands whose per-borrower disagreement `d(x)` **collapses off the policy vocabulary** onto un-named credit-risk features (`revol_util`, `inq_last_6mths`, `revol_bal`, `mort_acc`, …) on the subprime-ish tiers — `R²_named` 0.08–0.25 vs `R²_all` 0.51–0.57 on C2/C5/D4 (C1 recovers under used-feature-set de-dup; genuine collapsers C2/C5/D4). The interpretation in the regulator-doc spine is: *the documented 4-feature policy's vocabulary is inadequate to the model indeterminacy on those tiers* — the "rung 3" of the rung ladder (policy doesn't *name* the feature the admissible models fight over). The obvious objection: **that's a thin-demo artifact** — a real lender's underwriting policy names far more than four features; give the construction a serious policy and the gap closes. #11 tests that objection. It also — and this is the part with a real coin-flip prior — surfaces a refinement the LC result couldn't: on FM the *available* un-named features are largely **not credit-risk features a policy could legitimately name** (geography, lender identity, loan size), so a residual gap on FM would be **rung 3b** (the structure is real but a policy *shouldn't* name it — naming geography is redlining; lender identity isn't a borrower attribute), distinct from LC's **rung 3a** (the extension features are standard underwriting inputs the thin demo merely omitted — codification *could* close it). Two outcomes are both interesting and the prior between them is genuinely open; see §5.

This is also the first concrete instance of the codification layer that Shaun's brief #2 ("banking ontology woven through every layer") keeps asking for: the policy YAML built here is the first non-toy artifact in the schema, and building it locates where the `policy/encoder.py` micro-ontology generalizes (the monotonicity / mandatory / prohibited core probably transfers from the thin demo) and where it cracks (the Eligibility Matrix's hard cutoffs are *eligibility gates*, a constraint-type the current schema only fakes via `applicable_regime` — flagged below as a finding, not a blocker). See the ontology-design memory for the longer arc.

## 1. What gets built (before the run)

### 1a. The medium-fidelity policy YAML — `policy/fnma_eligibility_matrix.yaml`

A new policy graph in the existing `policy/encoder.py` schema (`{name, version, status, constraints:{monotonicity, mandatory_features, prohibited_features, applicable_regime}, nodes:[...]}`). **Status field: `medium-fidelity-reconstruction`** — and the file's header will say, in the same spirit as the thin demos' `ILLUSTRATIVE` banner: *this encodes the public structure of the Fannie Mae Eligibility Matrix (DTI / FICO / LTV-CLTV / occupancy / units / reserves cutoffs — values that are public and stable) and the standard Selling-Guide underwriting inputs; it is **not** a faithful transcription of the current Eligibility Matrix PDF (which is not in this repo) and not the policy of any specific lender's overlays.* The point is "a serious policy vocabulary," not "the literal document."

Contents (subject to what the FM collector can expose — §1b):

- **`mandatory_features`** (the underwriting inputs the Selling Guide names and an underwriter must consider): `fico_range_low`, `dti`, `ltv`, `cltv` *(if exposable; else omit)*, `loan_term_months`, `loan_purpose`, `occupancy_status`, `num_units`, `num_borrowers`, `first_time_homebuyer`, `property_type`, `mortgage_insurance_pct` *(if exposable)*, `amortization_type` *(if exposable; FM 2018Q1 is overwhelmingly fixed-rate so this may be near-constant — keep only if it has spread)*. Target ~10–14 mandatory features (vs the thin demo's 3).
- **`monotonicity`** (sign of effect on *grant* probability, the encoder's convention): `fico_range_low: positive`, `dti: negative`, `ltv: negative`, `cltv: negative` *(if present)*, `mortgage_insurance_pct: positive` *(more MI coverage → the lender is more willing to grant at a given LTV — direction stated relative to grant prob; revisit at build time if the data argues otherwise)*. Categorical features (`loan_purpose`, `occupancy_status`, `property_type`, `num_units`, `first_time_homebuyer`, `amortization_type`) are **mandatory but not monotonic** (no `monotonic_cst` entry — they appear in §1b's `is_feature_subset_admissible` gate, not in the sklearn monotonicity array).
- **`prohibited_features`**: `property_state` and `msa` (and `seller_name`, `servicer_name` if exposed) are **deliberately *not* prohibited here** — they are the extension features whose explanatory role the test measures (§3); a real policy graph *would* prohibit `property_state`/`msa` as redlining proxies, and the **comparison run** in §2 (variant B) does exactly that. (Recording this so the choice is explicit and pre-committed: the *primary* run leaves them admissible to *measure the gap*; the *comparison* run prohibits them to confirm the gap is them.)
- **`applicable_regime`**: `occupancy: owner_occupied` (matches the loader's filter), `loan_purpose: [purchase, cash_out_refi, no_cash_refi, refi_unspecified]`, `lien_position: first` (FM SF is first-lien only). Plus the Eligibility-Matrix *gates* expressed as the regime envelope where the standard automated path applies: `conforming_loan_limit: true` (loan amount within the 2018 baseline conforming limit — $453,100 for a 1-unit; higher for 2–4 units / high-cost areas — encode as a UPB ceiling, approximate, flagged), `fico_range_low >= 620`, `dti <= 50`, `ltv <= 97` (1-unit purchase) — out-of-envelope loans route to `manual_review` and are not scored (the encoder's existing semantics). **NB / finding to record in the result note:** these gates are currently smuggled into a free-text `applicable_regime` dict and a hand-written `nodes` graph; the schema has no first-class "eligibility gate" constraint type. #11 is the empirical demonstration that it wants one — a `gates:` block parallel to `monotonicity:` (feature, operator, threshold, on-fail-route). *Not building that schema extension in this test* (don't extend the schema on spec); recording that the test motivates it.
- **`nodes`**: a decision graph mirroring the thin-demo shape (entry → regime_check → conforming-limit gate → fico_floor → dti_ceiling → ltv_check → standard_grant; manual_review / decline leaves), expanded to reference the new gates. Human-readable; the encoder reads `constraints`, not `nodes`, so the graph is documentation + the audit artifact, as in the thin demos.

If a planned feature turns out not to be exposable from the 113-col layout without disproportionate effort (§1b), it is dropped from `mandatory_features` and the result note records exactly which — the policy is "as rich as the public data supports," and that ceiling is itself part of the finding (the FM public file is *thinner* than a real underwriting file: no income, no reserves balance, no liquid-assets, no residual-income — so even the "rich" FM policy is bounded above by the data, and a residual gap could partly reflect *data* thinness, not *policy* thinness; this caveat is load-bearing and goes in the result note's limits regardless of outcome).

### 1b. The FM collector extension — `wedge/collectors/fanniemae.py`

The collector already has `FIELD_POSITIONS` for `channel`, `num_borrowers`, `first_time_buyer`, `property_type`, `num_units`, `loan_purpose`, `occupancy`, `orig_interest_rate`, `loan_term`, `ltv`, `dti`, `credit_score`. Writes `runs/fm_rich_policy_vocab_adequacy_results.json`.

## 2. Setup (frozen here)

- **Data / vintages — the cross-vintage robustness arm is a *gate*, not a followup.** Primary: `data/fanniemae/2018Q1.csv` (mid-expansion). Robustness gate: also **2016Q1** (mid-expansion — "is the finding stable across a year within a regime?") and **2008Q1** (crisis onset — "is it stable across a different economic regime?"). All three already extracted on disk. Each loaded via `load_fanniemae(..., horizon_months=24)`; owner-occupied, purchase/refi, first-lien (the loader's eligibility filter); label = ever-90+-DPD within 24 monthly periods (loader default); one row per loan after collapse. Each quarter's collapse is ~12 min / ~30 GB RSS — **strictly serial, never two in parallel** (the project FM-load discipline; single-completion Monitor on the background load, not per-line). `nrows` large enough for stable band discovery — target the full owner-occ purchase/refi population per quarter (order ~3–5×10⁵ loans); if RSS bites, `nrows=1_000_000` monthly-rows then collapse (matches `run_wedge_fanniemae.py`), recorded. A random 50/50 *loan-level* split within each quarter (build on split 1, recompute `d`/explainers on split 2) is reported as a **bonus overfitting guard** — it is *not* the robustness arm; the three vintages are. (If the three disagree, that's when 2012Q1 / 2006Q1 / 2022Q1 get added — but ≥3 is the registered floor here.)
- **Stratum is a registered *factor* — two strata, parallel arms.** **(S-rate)** rate-band: `orig_interest_rate` cut into deciles **within each vintage** (`pd.qcut(rate, q=10, labels=False, duplicates="drop")`), labels `rb00`…`rb09` (rb00 = lowest rate ≈ best-quality, rb09 = highest rate ≈ the FM analogue of "subprime tier") — FM's published-pricing-tier analogue of LC's `sub_grade`, and what `scripts/run_pricing_fm.py` uses (the consistency argument: same kind of stratum as the LC work — both stratify on a pricing/grading *output*). **(S-llpa)** FICO×LTV grid cells: FICO bucketed `{<660, 660–679, 680–699, 700–719, 720–739, 740–759, 760–779, ≥780}` × LTV bucketed `{≤60, 60–70, 70–75, 75–80, 80–85, 85–90, 90–95, >95}` (≈ the Fannie LLPA grid granularity) — the *input-side* stratum (you haven't conditioned away the priced rate). S-rate is **primary**; S-llpa runs in parallel. Minimum 5 000 loans per stratum cell to be "analyzed"; smaller cells SKIPped and recorded. **The surprise to watch for:** a vocab gap present on S-llpa cells but *absorbed* on S-rate bands ⇒ "the priced rate is doing governance work the named policy isn't" — which loops straight back to the pricing finding ([[project_pricing_space_cat2]]). (Running both strata is also the methodology eating its own dogfood: S-rate = collapse-to-the-priced-tier, S-llpa = keep-the-input-stratification; doing both = the receipt for which collapse was warranted, per [[project_premature_collapse_frame]].)
- **Placebo / scrambled-policy baseline — the "vs what by chance?" anchor.** For each analyzed stratum cell, alongside the real medium-fidelity policy, build two control bands: **(C-random)** a *random* policy — `k` features picked uniformly at random as the "named" set (k = the number of exposed real-policy named features), no monotonicity constraints, R-majority enforcement on the random set; **(C-scrambled)** the *real* named feature set but with *reversed* monotonicity signs (FICO negative, DTI positive, LTV positive). Recompute `R²_named` for each control's named set against that control's own `d`. The real policy's `R²_named` is only meaningful relative to these: if real ≈ C-random, "the vocabulary is adequate" is hollow (any 12 features would explain `d` that well — `d` is just low-dimensional, not policy-aligned); if C-scrambled's R(ε) is empty or trivially small, that confirms the monotonicity constraints are binding (good — they exclude real alternatives). 5 random draws for C-random (seeds 1–5), report the mean and spread. Cheap (the controls reuse the same construction machinery).
- **Construction-hyperparameter sensitivity — the "is this an ε = 0.02 artifact?" anchor.** On the *headline* stratum cells only (the ones that hit a verdict criterion in §4 under the primary settings), re-run the band construction at **ε ∈ {0.01, 0.02, 0.05}** (and report whether the `R²_named` verdict — gap / no gap — is stable across them). Not a full grid sweep (depth and leaf-min are left at the registered values; ε is the one a reviewer fixates on); recorded as the sensitivity arm. If the verdict flips between ε = 0.01 and ε = 0.05, the result note says so and downgrades the claim accordingly.
- **Band construction per rate-band** (mirrors the extension-admitted / routable tests' construction): candidate feature set = **named-policy features** (the §1a `mandatory_features` ∩ what the collector exposes) **∪ extension features** (`property_state`, `original_upb`, `seller_name`, `servicer_name` — bucketed; `orig_interest_rate` excluded from the *candidate* set since it defines the stratum). `build_refinement_band(..., epsilon=0.02, depths={1,2,3}, leaf_mins={25,50,100}, seed=0, train_holdout=0.3)`; monotonicity `monotonic_cst` from the §1a policy enforced on the monotone named features only. De-dup **by used-feature-set** (the validated default — keep the highest-holdout-AUC representative per distinct used-feature-set; the runtime fix and the policy-meaningful notion of "distinct refinement"). Refit distinct members on the full band-population, freeze. `d(x)` = std over distinct refit members of P(default | x).
- **Mandatory-feature enforcement is a registered *factor*, not a single choice — three readings, all applied as post-hoc filters on the one within-ε construction** (so the construction runs once; the three readings are cheap). With ~12 mandatory features, depth-≤3 trees split on ≤7, so *no single shallow member can satisfy a 12-mandatory-feature requirement* — the per-member reading is incompatible with shallow trees, which is itself informative. The three readings: **(R-member)** every member's used-feature-set must include all mandatory features (the literal text-reading; the existing `is_feature_subset_admissible` semantics; the single-deployed-model-certification use case) — predicted to kill plurality entirely here (no shallow member qualifies); **(R-any)** the band's *union* of used features must cover the mandatory set (the institution-considers-these-features / process-level reading) — predicted near-vacuous at 12 mandatory features and bands of dozens of distinct feature-sets (some member almost always splits on each); **(R-majority)** each mandatory feature must be used by members covering ≥ 50% of the band population (the "the consideration actually bites" reading) — predicted the only one that's both satisfiable and non-trivial. The divergence between the three *is* a finding about the schema's `mandatory_features` mechanism (it does no work at either extreme; only the majority-reading constrains R(ε)). The headline geometry results (§ below) are reported under **R-majority as primary**, with R-member and R-any band sizes / plurality reported alongside; if R-member retains *any* plural bands they get the full treatment too.
- **Plurality** (same definition as #6 / extension-admitted): a rate-band is *plural* iff ≥ 3 distinct members AND ≥ 2 distinct used-feature-sets AND median pairwise ranking-Spearman among members < 0.9. The **analyzed-plural set** = rate-bands plural under this construction with ≥ 5 000 loans.
- **The geometry / vocab-adequacy measurement per analyzed-plural rate-band:** `disagreement_explainer` (regression CART, `max_depth ≤ 3`, `min_samples_leaf = 50`, 5-fold-CV R²) predicting `d` from (a) **named-policy features only** → `R²_named`; (b) **named ∪ extension** → `R²_all`; `ΔR²_ext = R²_all − R²_named`. Root-split feature and top-3 importances of each explainer. Univariate Spearman ρ(f, d) for every candidate feature. PD shape of `d` on the named-explainer root (`tails`/`threshold`/`monotone`/`flat`/`mixed`).
- **The rung-3a/3b classification per band where a gap remains** (`R²_named < 0.3` and `ΔR²_ext ≥ 0.15`): inspect *which* extension features carry the ΔR². **3a (codification-fixable)** iff the dominant extension contributors are features a mortgage underwriting policy could legitimately name and the public file merely lacks proxies for (none of the FM public extension set really qualifies — the FM file's extension columns are geography / lender / loan-size — so 3a on FM would itself be a surprise, e.g. if `original_upb` carries it and loan-size-as-risk is policy-nameable). **3b (codification-irreducible)** iff the dominant contributors are `property_state` / `msa` / `seller_name` / `servicer_name` — structure that's real (regional HPI dynamics; lender-level origination quality) but that a policy *shouldn't* (geography → redlining) or *can't* (lender identity isn't a borrower attribute) name. Mixed → reported as mixed with the breakdown.
- **Comparison run (variant B) — also a registered factor (`prohibit_geography ∈ {false, true}`).** Variant A (`false`): geography/lender admissible (to *measure* the gap). Variant B (`true`): `property_state`, `msa`, `seller_name`, `servicer_name` moved to `prohibited_features` — the realistic compliant policy. If variant A shows a gap carried by those features, variant B should *close* it (`R²_named ≈ R²_all`, because the band can no longer use them) — confirming the gap is them and that a *compliant* policy is, on FM, vocab-adequate-by-construction-but-blind-to-real-structure (the rung-3b reading: "complete" only because it's forbidden from naming what matters). The informative failure: B *doesn't* close it ⇒ a named-but-not-monotone categorical (`property_type`, `num_units`, `loan_purpose`) carries disagreement the depth-3 named explainer missed — a "rung 2.5".

## 3. The question, sharpened

For each analyzed-plural rate-band of FM 2018Q1, with R(ε) constrained by the medium-fidelity FNMA policy (~12 named features):

1. **Is `d` a legible function of the named policy features?** (`R²_named ≥ 0.3` ⇒ yes, the policy vocabulary is adequate to the indeterminacy on that band.)
2. **If not, what carries the residual disagreement?** — and is it rung 3a (codification-fixable) or 3b (codification-irreducible)?
3. **Does the compliant variant (B) close the gap?** — confirming (or not) that a *legally-clean* policy is "complete" only by being forbidden from naming what the unconstrained-extension members find.

## 4. Verdict criteria (pre-committed; do not move after seeing data)

Stated for the **primary configuration** — R-majority enforcement, S-rate stratum, variant A — and required to **replicate across all three vintages** (2018Q1, 2016Q1, 2008Q1) to count as a "verdict"; a result on 2018Q1 alone that doesn't hold on the other two is reported as *vintage-specific, not a finding*. The R-member/R-any, S-llpa, and variant-B arms are reported as cross-tabulated context (and the verdict is re-stated under S-llpa as a parallel headline if it diverges from S-rate — that divergence being the §2 "surprise to watch for"). All `R²_named` claims are read relative to the placebo baseline (real `R²_named` must exceed the C-random mean by ≥ 0.15 for "vocabulary adequate" to mean anything) and checked for ε-stability on headline cells.

- **"The gap closes" (the thin-demo-artifact verdict):** `R²_named ≥ 0.3` (and ≥ C-random mean + 0.15) on **every** analyzed-plural rate-band, **on all three vintages**. ⇒ rung 3 was a thin-demo artifact; a serious mortgage policy is vocab-adequate to the model indeterminacy; the regulator-doc "vocabulary inadequacy" claim must be downscoped to "the thin demo, not a real policy" and the C2/C5/D4 result reframed as "under-codification is a real but *fixable* failure mode."
- **"The gap recurs" (rung 3 is mechanism-intrinsic, not thin-demo-specific):** `R²_named < 0.3` **and** `ΔR²_ext ≥ 0.15` on **≥ 1** analyzed-plural rate-band, **replicated on ≥ 2 of the 3 vintages** (and the ≥1-band hit, ε-stable). ⇒ even a medium-fidelity policy underdetermines; report the 3a/3b breakdown; the regulator-doc claim stands and *strengthens* (it's not the thin demo). If the carrier is 3b (geography/lender) and variant B closes it: the sharpest statement — "a compliant policy is vocab-adequate only because it's forbidden the variables that carry the indeterminacy; the observability artifact is *most* needed exactly there, because there is no codification fix."
- **Ambiguous middle** (some bands `R²_named ≥ 0.3`, some `< 0.3` but `ΔR²_ext < 0.15`; or the verdict differs across vintages): report as quality-stratified and/or regime-dependent (which it will be — see §5; the pre-reg failure pattern says *predict the heterogeneity*) — name which rate-bands on which vintages fall where, and state the gap is real on band X / vintage Y and absent on band Z / vintage W. A clean "stable on 2016, different on 2008" outcome is itself a finding (regime-sensitivity of the vocabulary gap).
- **No analyzed-plural bands at all** (the rich policy + R-majority + de-dup leaves every rate-band with < 3 distinct used-feature-sets or median pairwise ρ ≥ 0.9, on a vintage): ⇒ a medium-fidelity policy *plus* used-feature-set de-dup so tightly pins R(ε) that there's no plural refinement set to measure — itself a finding ("a serious policy makes the refinement set near-degenerate; the indeterminacy was a thin-demo artifact of the policy being too weak to pin much"). Considered unlikely given the FM cat2-null result ("FM keeps all bands at full size" — large R(ε)); if it happens on all three vintages, it's the strongest version of "the gap closes."

## 5. Predictions, with priors (the bet — and where it'll most likely be wrong)

- **P1 — the gap recurs on the high-rate bands, closes on the low-rate ones.** Prior ~0.55 that ≥ 1 analyzed-plural rate-band hits the "gap recurs" criterion. *Why a coin-flip:* the rich FM policy names most of the legit underwriting risk features (FICO, DTI, LTV, MI, units, FTHB, purpose, occupancy) — so the within-rate-band residual disagreement, if any, has to be carried by geography / lender / loan-size, and it's genuinely unclear whether those carry ≥ 0.15 ΔR² *within a rate-band* (the rate-band already absorbs a lot of cross-loan risk variation; what's left within it may be near-noise the extension features can't explain either — in which case `R²_all` is also low and the "gap" is "nobody can explain `d`", which fails the `ΔR²_ext ≥ 0.15` leg). I'd back P1 hitting on **rb08–rb09** (highest-rate) if it hits anywhere.
- **P2 — if a gap remains, it's 3b (geography/lender), not 3a.** Prior ~0.65 (conditional on a gap). *Why:* the FM public file's extension columns just *are* geography/lender/size; there's no credit-risk extension feature on FM the way `revol_util`/`inq_last_6mths` were on LC. The interesting low-probability branch: `original_upb` (loan size) carries it — loan-size-as-risk is *partly* policy-nameable (conforming limits, jumbo overlays), so that'd be a 3a-ish wrinkle.
- **P3 — heterogeneity across rate-bands is the rule, not the exception** ([[project_pre_registration_pattern]] — the pre-regs that assumed uniformity missed; this one explicitly doesn't): the low-rate bands (rb00–rb03) are vocab-adequate (`R²_named ≥ 0.5`), the mid bands ambiguous, the high bands (rb07–rb09) the gap candidates. Prior ~0.7 that the result is a gradient by rate-band rather than uniform either way.
- **P4 — variant B closes whatever gap variant A shows.** Prior ~0.8 (near-mechanical: if the band can't use `property_state` etc., and they were the ΔR² carriers, `R²_all` drops to `R²_named` and the gap is arithmetic-closed). The informative failure: variant B *doesn't* close it ⇒ a *named-but-not-monotone categorical* (`property_type`, `num_units`, `loan_purpose`) carrying disagreement the depth-3 named explainer was missing — a "rung 2.5" (the policy *names* it but the explainer's functional form can't see it).
- **P5 — the real policy beats the random-feature placebo decisively.** Prior ~0.85 that real `R²_named` ≥ C-random mean + 0.15 on the prime/low-rate bands. *Why a bet, not a certainty:* if `d` is just intrinsically low-dimensional (a depth-3 tree on *any* 12 features explains it well because there are only ~3 directions of disagreement), real ≈ random and the "adequacy" framing partly collapses — that would be a genuinely deflating result for the whole vocabulary-adequacy line, and I'd want to know. C-scrambled's R(ε) should be near-empty or tiny (monotonicity binding) — prior ~0.7 (it could be that the data is monotone enough that the *reversed* constraint still admits a fair number of ε-good models, which would say the monotonicity constraints are mostly slack — itself a [[#3 dead-letter]] finding, see §6).
- **P6 — the headline verdict is ε-stable.** Prior ~0.7 that whatever §4 verdict the headline cells give at ε = 0.02 holds at ε = 0.01 and ε = 0.05. The plausible flip: at ε = 0.01 the bands shrink toward the named-feature core (the extension features get squeezed out as not-quite-competitive) → a gap at ε = 0.02 *closes* at ε = 0.01. If so the result note reports "the gap exists in the [0.02, 0.05] band-width regime, not the tightest one" — narrower claim, still real.
- **P7 — heterogeneity *across vintages*, not just across rate-bands.** Prior ~0.65 that the 2018Q1 and 2016Q1 results agree (within-regime stability) but 2008Q1 differs — most likely *more* gap on 2008Q1 (crisis-vintage loans default for reasons — regional HPI collapse, unemployment shocks — that no underwriting policy names, so the extension features [geography especially] carry more), or the opposite (crisis defaults are so dominated by the macro shock that *nothing*, named or extension, explains `d` well → `R²_all` low too → fails the `ΔR²_ext` leg). Either way, "the vocabulary gap is regime-sensitive" is the predicted shape, not "uniform across vintages." ([[project_pre_registration_pattern]] — this is the one I'm most confident *won't* be uniform.)
- **Cross-cutting most-likely-miss:** I'll have under-weighted the chance that the FM strata are just *too homogeneous within* to support a plural refinement set at ε = 0.02 once ~12 mandatory features (R-majority), used-feature-set de-dup, *and* the cross-vintage replication requirement are all imposed — i.e. the "no analyzed-plural bands" outcome (§4) is more likely than my ~0.1 gut estimate, because the thin-demo runs got their plurality partly *from the policy being weak*. If that's what happens, the honest read is "the gap was a thin-demo artifact" and the regulator-doc downscopes — a real result, not a null to walk back. Second-most-likely miss: 2008Q1's collapse/feature distributions are weird enough (huge default rates, truncated 24-mo windows for late-2008 originations) that it's not comparable and has to be dropped — in which case the robustness arm degrades to 2-vintage (2018+2016, both within-regime) and the cross-regime claim becomes a flagged followup, honestly stated.

## 6. What this does and doesn't claim, regardless of outcome (limits — in the result note verbatim)

- **The FM public file is thinner than a real underwriting file** (no income, no reserves, no liquid assets, no residual income, no manual-underwriter notes). The "medium-fidelity" policy is bounded above by the data. A residual gap (P1 hits) is consistent with *either* "real policies underdetermine" *or* "this particular public dataset is missing the features that would close it" — the result note must say both and not over-claim the first. The cleaner test of "real policies underdetermine" would be a real lender's data + real policy (out of scope; this is the best public proxy).
- **The policy YAML is a reconstruction of public structure, not the Eligibility Matrix PDF.** Numbers (DTI 50, FICO 620, LTV 97, conforming limit) are public and stable; the *exact* matrix cell structure (FICO×LTV grid for min-FICO-by-LTV, reserve months by occupancy/units) is approximated. This is the thin-demos' established standard ("public conventions inform the thresholds; specific values illustrative"), one notch up in fidelity.
- **Stratum choice (rate-band) is one defensible analogue of LC's `sub_grade`, not the only one.** FICO×LTV LLPA-grid cells are the other natural choice (it's literally how Fannie prices); rate-band is chosen for consistency with the existing FM pricing run. A followup could redo this on LLPA cells.
- **The schema doesn't have first-class eligibility gates.** #11 smuggles the Matrix cutoffs into `applicable_regime` + a hand-written `nodes` graph. The result note records this as the concrete motivation for a `gates:` constraint type — *not* built here.
- **No claim about case-level routing.** That door is closed ([[project_routable_population_result]], six ways). This test is purely about *observability* — whether a serious policy's vocabulary is adequate to the indeterminacy — at the tier (rate-band) level. If a gap recurs, it feeds the regulator-doc §B "vocabulary inadequacy" thread and the rung ladder; it does *not* reopen triage.
- **#11 is one cell of a larger structure: the documented-policy-vs-operative-behavior diff.** Project the documented policy D onto the operative behavior O ("map D back onto O" — Tony's framing) and the holes sort into three kinds — **dead letter** (D imposes a constraint O satisfies for free → the constraint is *slack*/decorative; a regulator reading D over-estimates the protection it provides), **unwritten policy** (O has structure D doesn't name → the C2/C5/D4 phenomenon), **distortion** (D's *shape* is wrong somewhere — says monotone-positive-on-FICO, O has a flat/reversed region). #11 is the **unwritten-policy cell, for mortgages**. The **dead-letter cell** is its own near-runnable test (a sibling pre-reg, *not* built here): add a "relaxed-constraint R(ε)" mode — drop one of D's constraints (e.g. monotonicity-on-FICO), reconstruct the ε-good set without it, check whether you *can* now fit an ε-good model that violates it; if yes the constraint was *binding*, if no it's a *dead letter* (the data already behaves that way; D's constraint is true-but-vacuous). The C-scrambled placebo (§2) is a first taste of it. The **distortion cell** is a third (fit the unconstrained ε-good models, check where they're non-monotone in features D says are monotone). The "hole report" — for each element of D: {present-and-binding / present-but-slack / present-but-distorted}, plus {O-structure-not-in-D} — is the codification deliverable's actual core; #11 is the first brick, the schema-extension experiment, and the ontology data-point all at once (see [[project_ontology_design_philosophy]]). This is recorded here so #11's result note frames itself as a *component*, and so the sibling pre-regs get written rather than rediscovered.
